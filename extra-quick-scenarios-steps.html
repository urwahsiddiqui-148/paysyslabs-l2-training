Step 1
Check Logs — find out why it went down
Do not restart blindly. Read the log first. Was it a crash? A disk full? A DB failure? The log tells you. If you restart without knowing why — it may crash again in 2 minutes.
tail -f /var/log/app.log
grep -i "error\|fatal\|crash" /var/log/app.log | tail -20
Step 2
Restart the Service — bring it back up
Once you know the cause and it is safe to restart — do it. Always use systemctl restart, not kill, for services. Get bridge approval before restarting on PROD.
systemctl restart app-service
systemctl status app-service
Step 3
Verify Health — confirm the restart worked
After restarting, confirm the service is actually healthy — not just "started" but accepting connections and responding correctly. Call the health endpoint.
curl -s https://your-app/health
systemctl is-active app-service
Step 4
Notify Team — close the loop
Update the bridge, send recovery message to Slack #incidents, update the Jira ticket with what happened, what was done, and the recovery time. Do not close a P1 without a post-incident update.
Jira: Service restored at [time]. Root cause: [cause]. Action taken: restart. Monitoring for recurrence.

Step 1
top — open the live view
Open top immediately. Press P to sort by CPU. The process at the top is your suspect. Note the PID and %CPU. Check the load average in the header — above 1.0 per core = overloaded.
top
cat /proc/loadavg
Step 2
Identify the process — confirm it is the problem
Use ps for a cleaner snapshot. Check how long the top process has been running. If a Java batch job normally takes 5 minutes but has run 3 hours — it is stuck.
ps aux --sort=-%cpu | head -6
ps -p PID -o pid,etime,pcpu,cmd
Step 3
Restart or kill — with bridge approval on PROD
Use kill -15 for a graceful stop. It tells the process to clean up and exit properly. Only use kill -9 if -15 does not work. Never kill -9 a database process.
kill -15 PID # graceful — try first
kill -9 PID # force — only if -15 fails
Step 4
Verify — confirm CPU recovered
After killing the process, watch the load average drop. Run ps again to confirm the process is gone. Check that transaction response times are back to normal.
cat /proc/loadavg # should decrease
ps aux --sort=-%cpu | head -4

Step 1
Check queue depth — note the number before touching anything
Check the current queue depth. Write the number down. This is your before-state. You need it to confirm messages were not lost after restart. Also check if the consumer service is running.
systemctl status payment-consumer
grep "Queue depth" /var/log/mq-consumer.log | tail -5
Step 2
Restart consumer — in the safe order only
The safe restart order is critical. Wrong order = lost messages. Stop consumer → get bridge approval → restart MQ if needed → start consumer. Never restart MQ while consumer is actively reading from it.
systemctl stop payment-consumer
systemctl restart rabbitmq-server # if MQ itself is stuck
systemctl start payment-consumer
Step 3
Monitor — watch the queue drain
After restarting, the queue depth should start decreasing as the consumer picks up messages. If depth stays high — the consumer restarted but is immediately failing again. Check logs.
watch -n 5 "grep 'Queue depth' /var/log/mq-consumer.log | tail -3"

Verify
df -h again
Step 1
df -h — confirm which partition is full
Run df -h to see all partitions. Find the one at 100%. Then use du to find which folder is the biggest consumer — it is almost always /var/log.
df -h
du -sh /var/log/* 2>/dev/null | sort -rh | head -8
Step 2
Clean logs — delete or move old log files
Delete log files older than a safe period (e.g. 30 days). Never delete the currently active log file — only rotated old ones. Confirm the application is still writing after cleanup.
find /var/log/app -name "*.log" -mtime +30 -delete
find /var/log/app -name "*.log.gz" -mtime +60 -delete
Step 3
Compress old files — archive instead of delete
For logs you need to keep (for compliance or investigation) — compress them with tar. A 500MB log file compresses to ~15MB. You keep the data but free 97% of the space.
tar -czf /backup/logs-$(date +%Y%m%d).tar.gz /var/log/app/old/
rm -rf /var/log/app/old/ # only after confirming backup
Step 4
Verify — confirm disk is healthy again
Run df -h again. Disk should be below 80%. Confirm DB is writing, logs are growing, and no write errors remain in the log file.
df -h # confirm below 80%
tail -f /var/log/app.log # confirm writes resumed

Step 1
curl — test the endpoint directly from the server
Use curl to call the failing endpoint from the server itself. This tells you the exact HTTP response code and error message. Also call /health to see if all dependencies are up.
curl -s -o /dev/null -w "%{http_code}" https://api/endpoint
curl -s https://api/health | python3 -m json.tool
Step 2
Logs — find the specific error behind the failure
Check the API service log for the error. A 502 usually means the downstream service is down. A 500 means the application code threw an exception. The log shows the exact error.
grep "ERROR\|500\|502\|503" /var/log/api.log | tail -20
grep -B 3 "ERROR" /var/log/api.log | tail -15
Step 3
Restart — if the process or dependency is down
If logs show the API process crashed — restart it. If logs show DB_CONNECTION_REFUSED — the DB is the problem, not the API. Fix the root cause, not the symptom.
systemctl restart api-service
systemctl status api-service
Step 4
Validate — confirm the endpoint is working again
Call curl again. You should get HTTP 200. Call /health and verify all dependencies show connected. Monitor logs for 5 minutes after restart to confirm no immediate re-failure.
curl -s -o /dev/null -w "%{http_code}" https://api/endpoint
tail -f /var/log/api.log # watch for new errors

✅ Quick Scenarios — What I Know

  • Service Down — check logs first to find why, then restart service, verify with curl /health, and notify the team with a Jira update
  • High CPU — open top, press P to sort, identify the heavy process by PID, check its etime, kill -15 with bridge approval, verify load average drops
  • Queue Backlog — check and note the queue depth first, stop consumer, get bridge approval, restart MQ if needed, start consumer, monitor depth decreasing
  • Disk Full — run df -h to confirm, use du to find the biggest folder, delete or compress old logs with tar -czf, verify disk is below 80% and writes resumed
  • API Failure — curl the endpoint for the HTTP code, check logs for the exact error, restart only if the process is the cause, validate with curl and monitor logs after
